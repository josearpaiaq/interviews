Explain the difference between goroutines and OS threads. How does the Go runtime scheduler (M:N model) work?
Go RuntimeConcurrency
Medium ▾
✦ IDEAL ANSWER
Goroutines are user-space threads managed by the Go runtime, starting at ~2 KB of stack (growing dynamically), while OS threads start at ~1–8 MB. The Go scheduler implements an M:N model: M goroutines multiplexed on N OS threads (Ps – logical processors).
  • G – goroutine (the unit of work)
  • M – OS thread (machine)
  • P – processor (holds a run queue, context for scheduling)
The scheduler uses work-stealing: idle Ps steal goroutines from busy Ps' run queues. Goroutines yield at I/O, function calls, channel operations, and runtime.Gosched().

When would you choose a buffered channel over an unbuffered one? Walk me through a real scenario from your experience.
ChannelsConcurrency Patterns
Medium ▾
✦ IDEAL ANSWER
Unbuffered: strict synchronization — sender blocks until receiver is ready. Good for signaling ("handshake") or ensuring the other side acknowledged.

Buffered: decouples producer/consumer up to the buffer size. Good for:
  • Worker pools – enqueue N tasks without blocking the dispatcher until workers catch up.
  • Rate limiting – token bucket pattern.
  • Batch ingestion – collect log events and flush periodically.
Candidate should mention the risk of goroutine leaks if a sender writes to a buffered channel but no consumer drains it, and how select + default or context cancellation prevents deadlocks.

How does Go handle memory management and garbage collection? What tuning knobs are available to a senior engineer?
MemoryGC
Hard ▾
✦ IDEAL ANSWER
Go uses a tri-color mark-and-sweep concurrent GC. Key points:
  • GC pauses are short (sub-millisecond) due to concurrent marking.
  • GOGC (default 100) controls heap growth ratio before next GC. Lowering it reduces memory at the cost of more frequent GC cycles; raising it does the opposite.
  • GOMEMLIMIT (Go 1.19+) sets a soft memory cap — crucial in containerized environments.
  • Escape analysis: allocations that escape to heap vs. stay on stack. Use go build -gcflags="-m" to inspect.
  • Reduce allocations: use sync.Pool for short-lived objects, prefer value types over pointers where appropriate, preallocate slices.

Describe idiomatic error handling in Go and how you'd design a multi-layer service (handler → service → repository) to propagate and wrap errors effectively.
Error HandlingClean Architecture
Medium ▾
✦ IDEAL ANSWER
Go treats errors as values. Senior engineers should mention:
  • fmt.Errorf("context: %w", err) for wrapping (Go 1.13+).
  • errors.Is / errors.As for inspection at call sites.
  • Sentinel errors (var ErrNotFound = errors.New("not found")) vs. custom error types.
  • In a layered service, the repository returns domain-level errors (e.g., ErrNotFound); the service wraps with context and may convert to gRPC status codes; the handler maps to HTTP/gRPC response. 
// Repository
if err != nil {
    return fmt.Errorf("userRepo.FindByID %d: %w", id, ErrNotFound)
}

// Service
if errors.Is(err, repo.ErrNotFound) {
    return nil, status.Errorf(codes.NotFound, "user %d not found", id)
}

How do you implement resilience patterns (circuit breaker, retry with backoff, bulkhead) in a Go service?
ResilienceDistributed
Hard ▾
✦ IDEAL ANSWER
  • Retry with exponential backoff + jitter: avoid thundering herd. Library: github.com/cenkalti/backoff.
  • Circuit Breaker: three states — Closed, Open, Half-Open. Library: github.com/sony/gobreaker.
  • Bulkhead: isolate resources per downstream — separate goroutine pools / semaphores per dependency using a buffered channel as a semaphore.
  • Timeout propagation: always use context.WithTimeout, never block indefinitely on an RPC call.
// Semaphore bulkhead
sem := make(chan struct{}, maxConcurrent)
sem <- struct{}{}  // acquire
defer func() { <-sem }()  // release

Hard ▾

Compare gRPC to REST. When would you choose one over the other for inter-service communication?
gRPCAPI Design
Medium ▾
✦ IDEAL ANSWER
gRPC advantages: strongly-typed contracts (Protobuf), HTTP/2 multiplexing, bi-directional streaming, smaller payload, built-in code generation, easy deadline/cancellation propagation.

REST advantages: human-readable, universal tooling, easier for external APIs and browser clients.

Choose gRPC for: internal service-to-service calls requiring high throughput, streaming (e.g., real-time feeds), polyglot environments.
Choose REST for: public APIs, webhooks, clients you don't control, simpler CRUD endpoints.

Mention gRPC-Gateway for exposing a REST façade over gRPC services.

How do you implement authentication and authorization in a gRPC service using interceptors?
InterceptorsSecurity
Hard ▾
✦ IDEAL ANSWER
gRPC interceptors are middleware — UnaryInterceptor for unary calls, StreamInterceptor for streams. Chain multiple with grpc.ChainUnaryInterceptor. 
func AuthInterceptor(ctx context.Context, req interface{},
  info *grpc.UnaryServerInfo, handler grpc.UnaryHandler,
) (interface{}, error) {
  md, _ := metadata.FromIncomingContext(ctx)
  token := md["authorization"]
  claims, err := validateJWT(token[0])
  if err != nil {
    return nil, status.Errorf(codes.Unauthenticated, "invalid token")
  }
  ctx = context.WithValue(ctx, claimsKey, claims)
  return handler(ctx, req)
}
  • Use mTLS for service-to-service transport security.
  • Use JWT in metadata for user identity.
  • Authorization logic lives in the service layer, not the interceptor.

How do you version a Protobuf API without breaking existing clients?
ProtobufAPI Versioning
Medium ▾
✦ IDEAL ANSWER
Protobuf is designed for backward/forward compatibility if you follow the rules:
  • Never reuse or change the numeric field tag of an existing field.
  • Adding new fields is safe — old clients ignore unknown fields.
  • Removing fields: mark as reserved to prevent tag reuse.
  • For major breaking changes, create a new package version (v2) — e.g., booking.v2.BookingService.
  • Use google.protobuf.FieldMask for partial updates to reduce coupling.

Describe how you'd implement server-side streaming in gRPC for a real-time event feed. What are the failure modes to handle?
StreamingReal-time
Hard ▾
✦ IDEAL ANSWER
// Proto
rpc WatchEvents(WatchRequest) returns (stream Event);

// Server implementation
func (s *Server) WatchEvents(req *pb.WatchRequest,
  stream pb.EventService_WatchEventsServer) error {
  for {
    select {
    case event := <-s.eventCh:
      if err := stream.Send(event); err != nil {
        return err // client disconnected
      }
    case <-stream.Context().Done():
      return stream.Context().Err()
    }
  }
}
Failure modes to handle:
  • Client disconnect: check stream.Context().Done() or Send() error.
  • Slow consumers: use a bounded buffer; drop or backpressure slow clients.
  • Reconnect logic: clients should implement exponential backoff reconnect with a cursor/offset to resume from last processed event.
  • Deadline exceeded: honor the RPC deadline set by the caller.

Walk me through how you'd diagnose and fix a latency spike in a production Go service.
Profilingpprof
Hard ▾
✦ IDEAL ANSWER
Systematic approach:
  1. Check distributed traces (Jaeger/Tempo) to isolate which span is slow.
  2. Check metrics: p99 latency per endpoint, downstream error rates.
  3. Enable net/http/pprof endpoint, capture CPU/heap/goroutine profiles.
  4. Analyze with go tool pprof -http :8080 cpu.prof — flame graph shows hot paths.
  5. Look for: lock contention (sync/mutex profile), excessive allocations (heap profile), goroutine pile-up (goroutine profile).
  6. Write a benchmark (testing.B) to isolate and measure the fix.
Common culprits: unbounded connection pools, missing DB indexes, JSON marshaling in hot paths (swap to Protobuf or easyjson), GC pressure from large allocations.

  • What is runtime/trace and how does it differ from pprof?

How would you design a caching layer for a high-read backend service? What cache invalidation strategies would you use?
CachingRedis
Medium ▾
✦ IDEAL ANSWER
  • In-process cache: sync.Map or ristretto — ultra-low latency, bounded memory. Good for small, frequently accessed, rarely changing data.
  • Distributed cache: Redis — shared across service replicas, supports pub/sub invalidation.
  • Cache-aside pattern: service checks cache → on miss, loads from DB → writes to cache.
  • Invalidation strategies: TTL (simple, may serve stale), event-driven invalidation (write-through / publish invalidation event on mutation), cache versioning.
  • Stampede protection: on a cold cache, many goroutines hit the DB simultaneously. Use singleflight.Group to coalesce concurrent requests for the same key.

How do you design a graceful shutdown for a Go service handling in-flight gRPC and async worker goroutines?
LifecycleReliability
Hard ▾
✦ IDEAL ANSWER
func main() {
  ctx, cancel := signal.NotifyContext(
    context.Background(), syscall.SIGTERM, syscall.SIGINT,
  )
  defer cancel()

  // Start server, workers...
  go server.Serve(lis)

  <-ctx.Done() // wait for signal

  // 1. Stop accepting new gRPC connections
  server.GracefulStop()

  // 2. Cancel context propagated to workers
  cancel()

  // 3. Wait for in-flight workers to finish
  wg.Wait()

  log.Info("shutdown complete")
}
Key points: signal.NotifyContext (Go 1.16+), GracefulStop waits for active RPCs, sync.WaitGroup for async goroutines, shutdown deadline with a second timeout context to avoid hanging forever.

System Design: Design a booking confirmation service for a travel company (like Unique Vacations). It must process ~10,000 bookings/day, send confirmation emails/SMS, and integrate with external vendor APIs. Walk me through your architecture.
System DesignAsyncgRPC
Hard ▾
✦ IDEAL ARCHITECTURE POINTS
  • Inbound: gRPC BookingService receives booking events from upstream services.
  • Async processing: publish to a message queue (Kafka/NATS) to decouple confirmation processing from the booking flow.
  • Workers: Go consumer goroutines read from queue — fan-out to email worker and SMS worker.
  • External vendor APIs: circuit breakers + retries; store vendor call results in a vendor_responses table for audit.
  • Idempotency: use booking_id as idempotency key — prevent duplicate sends on requeue.
  • Persistence: PostgreSQL for booking state; outbox pattern for event publishing.
  • Observability: metrics (confirmation success rate, vendor latency), traces per booking, alerting on DLQ (Dead Letter Queue) growth.
  • Scale: 10K/day ≈ ~0.1 TPS — well within a single service; design for 10x headroom.
